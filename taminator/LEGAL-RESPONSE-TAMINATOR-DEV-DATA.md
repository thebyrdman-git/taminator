# Response to Red Hat Legal: Customer Data on taminator.dev

## 1. Why / how was this data added to the public website?

**Short answer:** The data was included in **documentation examples** in the project’s MkDocs source. I **forgot to redact** that customer data before building and deploying the documentation site to GitHub Pages (which serves taminator.dev). As a result, those examples—real customer names, account numbers, and in one case a real contact name, phone number, and email—were published as part of the public site.

**Mechanism:**

- **taminator.dev** is served by **GitHub Pages** from the **gh-pages** branch of the repository (thebyrdman-git/taminator on GitHub).
- The site is generated by **MkDocs** (version 1.6.1). The workflow is: edit Markdown/docs in the repo → run `mkdocs build` → push the built output (HTML, assets) to the **gh-pages** branch → GitHub Pages serves that content at taminator.dev.
- The documentation included an “Intelligence – How It Works” section intended to explain how the product extracts case numbers, customer identification, and contact information from emails. To illustrate that, the **source docs used real-world-style examples**, including:
  - Customer/domain names (e.g., Wells Fargo, JP Morgan Chase, Bank of America).
  - A real Red Hat account number (e.g., 334224) in an example.
  - In the “Step 2: Contact Extraction” example, a **real contact**: name, title, organization, phone number, and email address (appearing to be an actual customer contact), used to show how the product parses email signatures.

There was **no separate “customer data upload” or database** behind the site. The sensitive content was **embedded in the documentation source** as examples. When the MkDocs build was deployed to gh-pages, that content was included in the built HTML and thus became publicly available at URLs such as:

`https://taminator.dev/intelligence/how-it-works/#step-1-pattern-extraction`  
(and the “Step 2: Contact Extraction” section containing the contact details).

**Root cause:** I forgot to redact the customer data from the documentation before deploying. The docs used real (or real-looking) examples for clarity; I should have replaced them with placeholders or generic examples before publishing the site to gh-pages but did not do so. The standard MkDocs → gh-pages deployment then published the unredacted content.

## 2. For how long was that data available on the website?

**Answer:** From the **first deployment of the MkDocs-built site to gh-pages** until the site was put into **maintenance mode** (replacing all content with a single maintenance page).

Based on the **gh-pages** branch history:

| Event | Date (commit timestamp) |
|-------|--------------------------|
| **First deployment** of the full documentation site (including the “How It Works” page with the examples above) to gh-pages | **November 11, 2025** at 14:42:59 (EST) |
| **Removal** of site content and switch to maintenance-only page | **March 6, 2026** at 09:33:46 (EST) |

So the data was available on the public website for approximately **four months** (about 115 days): from **November 11, 2025** through **March 6, 2026**.

---

## Summary for Legal

- **How it happened:** Real customer/contact data was used in documentation examples in the MkDocs source. I forgot to redact that data before deploying to GitHub Pages; the deployment then published the unredacted docs on taminator.dev. No intentional leak—an oversight before publish.
- **Duration:** From first deploy (**November 11, 2025**) until maintenance mode (**March 6, 2026**) — about four months.
- **Remediation taken:** The site was put into maintenance mode; all previous content was removed from the deployed branch. Only a maintenance message is now served. (Note: Search engine caches and git history may still hold the old content; removal/redaction requests and optional history cleanup can be addressed separately if required.)
