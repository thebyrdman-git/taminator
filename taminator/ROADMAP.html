Taminator Roadmap

Taminator is by TAMs, for TAMs. Roadmap items are driven by TAM feedback; suggestions are welcome and will be evaluated. If your organization has the roadmap on GitLab, a link may appear at the bottom of this page.

New features

This section describes new features and enhancements recently added to Taminator.

  • Report formats — Reports can be generated and exported as Markdown, HTML, plain text, and CSV (e.g. for Google Sheets). Format is selectable in Report Manager when building or customizing the template; Check/Update produces the report in the chosen format. CSV columns align with the report table (case, summary, status, JIRA link).
  • Roadmap in app — This roadmap is shown inside the app via the Roadmap item in the top nav. An optional link to the roadmap on GitLab can appear at the bottom of this page if configured for your environment.
  • Gmail draft — From Check/Update Reports (after running Check or Update) and from Library (per-report actions), a Create Gmail draft button sends the current report body to Gmail as a new draft. The body is converted from markdown to HTML with Google Docs–style inline formatting (Arial, heading sizes, list indentation) so it looks like content pasted from Docs, not raw markdown. Requires Google connected in Settings (same OAuth as Drive/Docs).
  • Google Drive & Docs — In Settings, one-time setup: add a Google Cloud Client ID and Secret, then click Connect Google to complete OAuth. After that, from Check/Update Reports and Library you can open the current report as a new Google Doc, and use Back up to Google Drive to upload all report files to a new folder. Credentials are stored in ~/.config/taminator/; the User Guide has step-by-step instructions for obtaining the Client ID and Secret.
  • Trimmed UI copy — Report Manager, Check/Update Reports, Library, and Settings panels show only short introductory text and action buttons. Long instructions (e.g. how to get tokens, how to set env vars, how to get a Google Client ID) were moved to the User Guide. Each panel includes “See User Guide: [section]” links that open the User Guide panel and scroll to the relevant section.
  • User Guide — One in-app panel (User Guide in the nav) holds all extended documentation. It is built from the repo’s user-guide markdown (e.g. USER-GUIDE.md or README + GETTING-STARTED). Other panels link to it using #guide-<section-id> so the guide opens and scrolls to the right heading. Keeps the UI minimal while keeping full instructions one click away.
  • Refresh behavior — The app uses the URL hash (e.g. #reports, #config, #roadmap) to remember the current panel. When you refresh the browser (F5 or reload), the same panel is shown again instead of resetting to the default. Hash is updated when you switch panels via the nav.
  • Red Hat look and feel — The entire UI uses Red Hat design tokens (brand red, canvas, text, borders, success/error states). Links are styled with Red Hat red, not blue. Buttons, nav, cards, and status indicators use the same palette. Gmail draft bodies use Docs-style inline styles (font family, sizes, margins) so emails match the look of content copied from Google Docs into Gmail.

JIRA Cloud integration (plan)

Goal: Let Taminator work with JIRA Cloud instances (e.g. customer or partner Atlassian sites) in addition to Red Hat’s JIRA (issues.redhat.com).

Current state: The app is built for Red Hat JIRA only: base URL https://issues.redhat.com/rest/api/2, Bearer token auth, used for check/update (issue status) and report generation.

Planned approach:

  1. Configurable JIRA base URL
    Add a setting (e.g. in Settings or ~/.config/taminator) for “JIRA URL” (default: https://issues.redhat.com). For JIRA Cloud, user sets e.g. https://yourcompany.atlassian.net. API path remains /rest/api/2 (or configurable /rest/api/3 if needed).
  2. Auth for JIRA Cloud
    Red Hat JIRA: keep current Bearer token (Personal Access Token). JIRA Cloud: use Atlassian API token (email + API token). Optional: store “JIRA type” (redhat | cloud) or infer from URL (*.atlassian.net → Cloud). For Cloud, UI/CLI collects email + API token; for Red Hat, keep existing token flow.
  3. API compatibility
    Use same REST endpoints where possible: /rest/api/2/issue/{key}, fields status, summary, assignee, updated. JIRA Cloud’s REST API is largely compatible; any field or path differences handled with small conditionals or a thin adapter.
  4. Browse links
    Today: https://issues.redhat.com/browse/{id}. For Cloud: https://<tenant>.atlassian.net/browse/{id}. Generate from configured base URL.
  5. Phasing
    Phase A: Add configurable base URL + browse link generation (no new auth yet); keep Red Hat token.
    Phase B: Add JIRA Cloud auth (email + API token) in Settings and CLI; when URL is Cloud, use that auth. Same check/update and report flow.
    Phase C: Any extra Cloud-only features (e.g. Cloud-specific fields or permissions) as needed.

Out of scope for initial plan: OAuth 2.0 for JIRA Cloud (optional later); multiple JIRA instances in one run.

Detailed steps, env vars, and file changes are in docs/JIRA_CLOUD_INTEGRATION.md.
