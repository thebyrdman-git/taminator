# Taminator v2.0.1 — Release notes and user testing guide

**Release date:** TBD  
**Previous release:** v2.0.0

This release adds JIRA Cloud support, encoded credentials, optional HashiCorp Vault integration, and audit cleanup (generic defaults, no hardcoded contacts).

---

## What's new

### JIRA Cloud
- Use **JIRA Cloud** (e.g. `https://your-tenant.atlassian.net`) in addition to Red Hat JIRA (issues.redhat.com).
- In **Settings → Token configuration → JIRA instance**, set **JIRA base URL**. For Cloud (`*.atlassian.net`), enter your **Email** and **API token** (create at [id.atlassian.com/manage-profile/security/api-tokens](https://id.atlassian.com/manage-profile/security/api-tokens)). Check, Update, and report links use the configured instance.
- **In-app guidance** matches the instance: when base URL is Cloud, "How to obtain" points to the Atlassian API token page; when Red Hat, to issues.redhat.com Personal Access Tokens. Same in `tam-rfe config --add-token`. See User Guide → JIRA Cloud.

### Credentials and secret storage
- **Encoded token storage:** JIRA, Portal, and Hydra tokens saved from Settings are stored in `~/.config/taminator/ui_tokens.json` as a **base64-encoded payload** (not plaintext). File mode 0600. Existing plain JSON is still read; the next save upgrades to encoded format.
- **Optional HashiCorp Vault:** When **VAULT_ADDR** and **VAULT_TOKEN** are set, Taminator can use Vault as the secret store. Tokens are read from Vault when available (fallback to local file), and saving a token from Settings syncs it to Vault. Optional: **VAULT_NAMESPACE**, **VAULT_CACERT**, **VAULT_SKIP_VERIFY**. **tam-vault** CLI and the in-app Vault tab are unchanged. Setup: docs/VAULT_INTEGRATION.md.

### Bug fixes and behavior
- Token saving writes to the local encoded file and, when Vault is configured, to Vault (web server and desktop app).
- Server startup: improved PATH/Python resolution; error dialog shows server stderr and a Terminal workaround.
- Vault client: no default server; **VAULT_ADDR** required. SSL uses **VAULT_CACERT** and **VAULT_SKIP_VERIFY** when set.

### Audit cleanup
- Default TAM name and contact are blank or "TAM"; set your own in Settings or onboarding.
- Help and feedback point to GitLab issues. Onboarding and templates use generic names.

### Platforms
- **Linux:** AppImage (x86_64 and ARM64). On Fedora you may need `sudo dnf install zlib` if you see `libz.so` errors.
- **macOS:** DMG (Intel and Apple Silicon). First open: right-click → Open if Gatekeeper blocks.
- **Windows:** NSIS installer (x64).

## User testing steps

### 1. Basic run (no new config)
- Install or replace with the v2.0.1 build for your OS.
- Open the app; confirm it starts and you can open Reports, Library, and Settings.
- Run **Check** or **Update** on a report that already has JIRA IDs (Red Hat JIRA).
- **Pass:** App runs and Check/Update work with your existing token.

### 2. Encoded credentials
- In **Settings**, change or re-save one token (e.g. JIRA or Portal).
- Confirm `~/.config/taminator/ui_tokens.json` is not human-readable (base64-style content).
- Run **Check** or **Update** again.
- **Pass:** Token still works and the file remains encoded.

### 3. JIRA Cloud (if you have a Cloud instance)
- In **Settings → Token configuration → JIRA instance**, set **JIRA base URL** to `https://<your-tenant>.atlassian.net`.
- Confirm the **JIRA Cloud** section appears; enter **Email** and **API token**, then **Save JIRA settings**.
- Run **Check** or **Update** on a report that has issues from that Cloud instance.
- Open a JIRA link in the report; confirm it goes to your Cloud URL.
- **Pass:** Cloud auth works and links open in the correct instance.

### 4. JIRA "How to obtain" guidance
- With base URL set to Red Hat (or blank): run `tam-rfe config --add-token`, choose JIRA; confirm instructions point to issues.redhat.com.
- With base URL set to `*.atlassian.net`: same flow; confirm instructions point to id.atlassian.com API tokens.
- **Pass:** Instructions match the JIRA type you configured.

### 5. Optional — HashiCorp Vault
- Set env: **VAULT_ADDR**, **VAULT_TOKEN** (and optional **VAULT_NAMESPACE** / **VAULT_CACERT** if needed).
- Start the app; in Settings, save a token (e.g. JIRA or Portal).
- Confirm the token works for Check/Update (or Portal). If you have Vault UI/CLI, confirm the token appears in Vault.
- **Pass:** Token saves and works; optionally visible in Vault.

### 6. Upgrade from v2.0.0
- If you had v2.0.0, replace with v2.0.1 without deleting config.
- Open the app; run Check or Update.
- **Pass:** No re-login; existing tokens still work.

### 7. Platforms
- **macOS:** First launch — right-click → Open if Gatekeeper blocks.
- **Linux (Fedora):** If you see `libz.so` errors, run `sudo dnf install zlib` then retry.
- **Pass:** App launches and runs as expected.
